# 第一轮测试修复 — 设计文档

日期：2026-06-03
依据：`第一轮测试结果 spec.md`、产品 Spec、当前源码核实
范围：第一轮人工测试发现的 15 个问题（P0×1 / P1×6 / P2×8）+ §9 LLM prompt 加固（豆包 Mini2.0）

## 1. 目标与决策

把第一轮测试发现的全部问题修掉，使核心流程可用、AI 上下文完整、数据可沉淀，并统一展示文案/格式。本轮一次性覆盖全部 15 项 + §9 prompt 加固，下轮直接进入人工复测。

已确认的规划决策：

- **切分策略**：按优先级分批（符合测试 spec 1.3 推荐顺序），共 7 个 task，严格按序执行。
- **覆盖范围**：全部 15 项 + §9 prompt 加固（§9.1 已判定本轮采纳）。
- **展示语义**（spec 留给产品确认的部分）：
  - 已结束活动出勤概况显示 `X 实到 ｜ Y 应到`（数字在前）。
  - 已取消活动出勤概况显示 `—`。
  - 活动建议持久化采用单一 `advice`（JSON 字符串）字段 + `adviceUpdatedAt`（活动 type 固定，不拆训练/比赛两字段）。

## 2. 源码核实结论（与测试 spec 的差异）

逐条对照源码后，spec 的现象与定位基本准确，以下几处需在实现时注意：

- **TP-ATT-01（P0）**：双重根因——① 后端对非 `ended` 活动返回 409（`attendance/routes.ts:15`，行为正确）；② 前端 `AttendanceTab.mark()` 无 try/catch，而 `api.ts` 在非 2xx 时 `throw`，错误被静默吞掉；③ 出勤 Tab 在任何状态都渲染按钮。需同时按状态控制按钮 + 补错误反馈。`AttendanceTab` 当前 `Detail` 类型缺 `status` 字段。
- **TP-MEM-04（P2）**：`style`（风格）枚举**已经是中文**（进攻型等），只有 `position`/`level` 需要英→中映射。
- **TP-REV-01（P1）**：复盘 prompt 用的 `memberLine` 默认带 `captainNote`，会把队长备注泄进复盘 prompt，须一并处理（§9.4）。
- **§9 prompt 加固**：很多约束源码里**已部分存在**（memberBot 边界、scenarios 的 JSON 约束与"建议/可考虑"措辞、`summaryStage` 字段已落库并写入）。本轮是增量加固 + 补 prompt 断言测试，不是从零写。
- 测试基建齐全：后端 `scenarios.test.ts` / `memberBot.test.ts` / `assistant.test.ts` 可用 fake LLM 捕获 prompt；前端 `web-admin/test/*.tsx` 覆盖各页。

## 3. Task 分解（按序执行 T1 → T7）

### T1 ·〔P0〕出勤标记可用 + 错误反馈

- **覆盖**：TP-ATT-01
- **改动**：
  - `web-admin/src/pages/tabs/AttendanceTab.tsx`：`Detail` 类型补 `status`；按 `status==="ended"` 控制"标记已到场/未到场"按钮——非 ended 时隐藏或禁用并给明确提示；`mark()` 包 try/catch，失败时 Toast。
  - `web-admin/src/pages/ActivityDetail.tsx`：确认把 `status` 传给 `AttendanceTab`（当前已传完整 `detail`）。
  - 后端 `attendance/routes.ts` **不改**（409 对非 ended 是正确约束）。
- **验收标准**：
  - ended 活动点击"标记已到场"→ 行内"实际到场"显示"已到场"；"标记未到场"→ 显示"未到场"。
  - 非 ended 活动不出现"可点击但无反应"的按钮（隐藏/禁用/提示其一）。
- **测试**：`web-admin/test/attendance.test.tsx` 覆盖 ended 成功更新 + 非 ended 明确禁用/提示。

### T2 ·〔P1〕AI 上下文补全 + 对应 prompt 边界/JSON 约束

- **覆盖**：TP-ACT-02、TP-REV-01、TP-FT-01 + §9.2 / §9.3 / §9.4 / §9.5
- **改动**：
  - `ai/scenarios.ts` `generateMatchAdvice`：user prompt 补 `预计时长：{durationMinutes}分钟`、`活动类型：比赛`、参加人员名单规则；system prompt 加强"不得把未在名单中的队员排入首发或替补"，保留"建议/可考虑"。
  - `ai/scenarios.ts` `generateReviewSummary`：user prompt 补 `【注意事项】{a.notes ?? "未填写"}`；到场队员改用 `memberLine(m, false)`（已有 `includeNote` 参数），从源头不注入 `captainNote`；system prompt 明确面向全员公开、不得输出队长备注原文/复盘原文/录音转写原文/内部内容。
  - `ai/memberBot.ts` `answerMemberQuestion`：最近公开摘要改为 fallback——优先 `review.aiSummary`，否则 `activity.summary`，都无再传"暂无"（复用 `context.ts` 已有范式）；下一场活动补活动类型；system prompt 收紧公开/越界边界；不注入 captainNote / rawNotes / 后台助理上下文。
  - 上述三个场景的 system prompt 各自补 §9.2 全局约束（只依据给定数据、缺信息写"未提供/信息不足/暂无"、只返回 JSON 对象不带 Markdown/解释、字段严格匹配 schema）。
- **验收标准**：
  - `generateMatchAdvice` 传给 LLM 的 user prompt 含 `durationMinutes` 对应分钟数与参加人员名单规则。
  - `generateReviewSummary` user prompt 含活动注意事项，且不含队长备注原文。
  - memberBot：最近 ended 活动只有 `summary` 无 `aiSummary` 时 prompt 含该 `summary`；两者都有时优先 `aiSummary`；prompt 不含 captainNote/rawNotes/后台历史。
- **测试**：`scenarios.test.ts`（捕获 match/review prompt 断言）、`memberBot.test.ts`（新增 `activity.summary` fallback 覆盖）。

### T3 ·〔P1〕活动建议生成结果持久化

## 4. 依赖与顺序约定

1. **严格 T1 → T7 顺序**：P0 先解锁核心流程。
2. **T3 是本轮唯一改 schema 的 task**（一次 Prisma migration）。
3. **§9.2 全局 JSON 约束按场景归属拆分**：match/review/memberBot 在 T2，training/summary/assistant 在 T7，每个 prompt 字符串只编辑一次。
4. **T2 与 T7 都动 `ai/scenarios.ts` / `assistant/service.ts`，但函数不重叠**（match/review/memberBot ↔ training/summary/assistant），按序执行无冲突。
5. **T5 改 `memberLine` 为中文 label** 会改变 T2/T7 已写的 AI prompt 字符串 → T5 完成后回归这两个 AI 测试套件确认仍绿。
